Specialized Instrumental Dental — Documentación Técnica

v2.0 — Abril 2026

Análisis Exhaustivo del Sistema

Documentación técnica integral de la arquitectura MVC en 3 capas, flujos de datos, diccionarios de funciones y estructura de eventos del sistema Specialized.

1. Resumen Ejecutivo

Specialized es un sistema web de gestión de inventario e instrumental dental que sigue una arquitectura MVC en 3 capas:

CapaTecnologíaResponsabilidad
DatosPostgreSQL + Funciones AlmacenadasToda la lógica CRUD, validaciones de negocio, triggers de auditoría
BackendPHP 8 + PDOEnrutamiento API, autenticación JWT/Sesión, capa de acceso a datos
FrontendJavaScript Vanilla (SPA parcial)Renderizado dinámico, formularios, validación, navegación sin recarga
ℹ️ Importante
Principio fundamental del sistema: Ningún archivo PHP escribe SQL crudo inline. Todo pasa por querys.php que a su vez llama funciones almacenadas de PostgreSQL (fun_insert_*, fun_update_*, fun_delete_*). Esto centraliza la lógica de datos en un solo punto.

¿Qué hace? Ejecuta 9 subconsultas en paralelo y retorna UN SOLO objeto JSON con todas las listas de referencia del sistema.

¿Cuándo se ejecuta? Cada vez que se abre un formulario de gestión (crear/editar usuario, cliente, empleado, instrumento, etc.).

¿Qué retorna?

4.3 fun_obtener_stats() — Métricas del Dashboard

¿Qué hace? Calcula 16 métricas agregadas del sistema en una sola consulta.

¿Cuándo se ejecuta? Al cargar menuPrincipal.php (dashboard del admin).

¿Qué retorna? total_usuarios, usuarios_mes, total_admins, total_clientes, total_productos, total_instrumentos, total_kits, mayor_venta_nombre, mayor_venta_total, menor_stock_nombre, menor_stock_cant, total_ventas, ventas_diarias, total_proveedores, alerta_stock_critico, val_pordesc.

4.4 fun_obtener_alertas_detalle() — Stock Bajo

¿Qué hace? Busca todos los instrumentos, kits y materias primas cuyo cant_disp <= stock_min.

¿Cuándo se ejecuta? Al cargar el dashboard, para mostrar el badge rojo de notificaciones.

5. CAPA 2 — Backend PHP (API + Modelo)

5.1 api_gestion.php — El Enrutador Central

ℹ️ Importante
Este es el ÚNICO punto de entrada para toda la API del admin. Toda petición JS pasa por aquí.

Flujo interno paso a paso:

1. Incluir dependencias (security_headers, querys, jwt, controladores) 2. Header JSON + no-cache 3. Verificar método HTTP (solo GET y POST) 4. Verificar sesión PHP → $_SESSION['id_usuario'] 5. Si es POST → verificar rol admin → $_SESSION['id_menu'] == 1 6. Crear instancia de CQuerys → $db = new CQuerys() 7. Obtener conexión → $conn = $db->getConn() ═══ SI ES GET ═══ 8. Leer $tipo = $_GET['tipo'] 9. Ejecutar handleGet() de CADA controlador en cadena: → ApiDashboardController::handleGet() // auxiliares, stats, clientes, empleados, proveedores, usuarios, finanzas... → ApiInstrumentalController::handleGet() // instrumentos, kits, kit_instruments → ApiMatPrimController::handleGet() // categorias_materia, materias_x_cat, historico_precios... → ApiProductosController::handleGet() // productos → ApiHistorialController::handleGet() // kardex, ventas, facturas... 10. Si NINGUNO hizo exit → "Tipo no válido" ═══ SI ES POST ═══ 8. Detectar si es FormData o JSON puro 9. Si hay archivo (img_url) → moverlo a /images/{subFolder}/ 10. $conn->beginTransaction() 11. Switch por $accion: 'create' → delegar a handleCreate() de cada controlador 'update' → delegar a handleUpdate() de cada controlador 'delete' → ejecutar handleDelete() de cada controlador + deleteGeneric() 'restore'→ ejecutar handleRestore() de cada controlador 12. $conn->commit() o $conn->rollBack() si hay error

Patrón de delegación en cadena (Chain of Responsibility)

PHP// Para CREATE:
$res = ApiHistorialController::handleCreate($db, $tipo, $data);
if ($res === false) $res = ApiDashboardController::handleCreate($db, $tipo, $data);
if ($res === false) $res = ApiInstrumentalController::handleCreate($db, $tipo, $data, $parsedInput);
if ($res === false) $res = ApiMatPrimController::handleCreate($db, $tipo, $data);
if ($res === false) throw new Exception("Tipo no soportado");

Cada controlador retorna false si no le corresponde el $tipo, y el router prueba el siguiente.

Manejo de archivos (imágenes)

Cuando el formulario incluye un archivo ($_FILES['img_url']):

  1. Determina la subcarpeta según el tipo: instrum/, kit/, prod/, mat_prima/
  2. Valida extensión (solo JPG/PNG)
  3. Genera nombre único con md5(time() . $fileName)
  4. Mueve el archivo a images/{subFolder}/
  5. Guarda la ruta relativa en $data['img_url']: '../../images/instrum/abc123.jpg'

Patrón de cada método INSERT

PHPpublic function insertInstrumento($d) {
    // 1. Validar precondiciones (ej: imagen obligatoria)
    if (empty($d['img_url'])) throw new Exception("...");
    
    // 2. Preparar sentencia que llama a la función SQL
    $sql = "SELECT fun_insert_instrumentos(:id_espec, :nom, ...) as result";
    $stmt = $this->conn->prepare($sql);
    
    // 3. Ejecutar con parámetros sanitizados
    $stmt->execute([':id_espec' => (int) $d['id_especializacion'], ...]);
    
    // 4. Retornar resultado
    return $stmt->fetch(PDO::FETCH_ASSOC); // → ['result' => true/false]
}

Patrón de deleteGeneric()

1. Switch por $tipo → determina qué función SQL llamar (fun_delete_instrum, etc.) 2. Verificar que el registro EXISTE (SELECT ind_vivo FROM tab_X WHERE id = :id) 3. Verificar que NO fue eliminado previamente (ind_vivo = false → error) 4. Ejecutar la función SQL de borrado 5. Retornar resultado

5.4 conexion.php — Conexión PDO

1. Lee credenciales de .env (DB_HOST, DB_NAME, DB_USER, DB_PASSWORD, DB_PORT) 2. Verifica que las variables existan 3. Verifica que PHP tenga extensión pdo_pgsql 4. Crea conexión PDO con modo ERRMODE_EXCEPTION 5. Retorna $conn (o null si falla)

5.5 validar.php — Login

1. Recibe POST con usuario + password 2. Busca usuario en tab_users JOIN tab_users_menu 3. Verifica contraseña: - Si es hash moderno → password_verify() - Si es texto plano (legado) → compara + migra automáticamente a bcrypt 4. Lee ind_idle de tab_parametros (minutos de inactividad) 5. Genera JWT con payload: {id, usuario, menu, iat, exp} 6. Retorna {success, token, id_menu}

6.5 logica.js — Motor de Formularios Genéricos (1520 líneas)

Este archivo es el corazón de la generación dinámica de formularios. Sus funciones clave:

cargaAuxiliares() — Carga de datos auxiliares

1. fetch('./api_gestion.php?tipo=auxiliares') 2. Parsea JSON 3. Guarda en window.auxiliares → RESULTADO: window.auxiliares = {ciudades:[...], documentos:[...], ...}

abrirGestion(tipo) — Abrir modal de gestión de una entidad

1. Muestra modalGestion con spinner 2. En paralelo: fetch(datos de la tabla) + cargaAuxiliares() 3. Guarda datos en variable currentData 4. Llama renderTabla(data, columns, tipo) 5. Muestra botón "Nuevo" si no es 'nuevos_mes'

¿Desde dónde se llama? Desde las tarjetas del dashboard: onclick="abrirGestion('usuarios')", onclick="abrirGestion('clientes')", etc.

mostrarFormulario(tipo, accion, data, originType) — Genera formulario dinámico

1. Si no hay auxiliares → cargaAuxiliares() (lazy load) 2. Si tipo === 'kits' → renderiza Kit Builder especial (2 columnas, selector de instrumentos) 3. Si no → obtener esquema del formulario: obtenerEsquemaFormulario(tipo, data) → Esto busca en window.FormSchemas[tipo](data) 4. Generar HTML: recorre cada campo del esquema: - type='select' → <select> con options de auxiliares - type='datalist' → <input> + <datalist> + <input hidden> para ID - type='file' → <input type="file"> - type='number' → convierte a <input type="text"> con formato de miles - type='text' → <input> con pattern/maxlength 5. Inyectar HTML en modalGestionBody 6. Inicializar validaciones en tiempo real: initFormValidations(form)

guardarRegistro(event, tipo, accion, id, returnType) — Enviar datos al servidor

1. event.preventDefault() 2. Recolectar datos del formulario: - Si hay file → usar FormData - Si no → usar objeto JSON 3. Para campos number: limpiar formato de miles (quitar puntos) 4. Para kits: recolectar instruments[] del Kit Builder 5. Llamar ApiService.fetchApiGestion(accion, tipo, payload, id) 6. Si éxito → mostrar SweetAlert + recargar datos: - Si origin='local' → refreshInstrumentalTable(tipo) (para instrumental) - Si origin='local_materia' → refreshMateriaPrimaView() (para materia prima) - Si no → abrirGestion(returnType) (para dashboard) 7. Si error → mostrar mensaje de error del servidor

6.6 Los FormSchemas — Definición de Formularios

Cada controlador JS registra sus esquemas de formulario en window.FormSchemas:
